# doc/tasks/ — ระบบติดตาม Task งาน

Last Updated: 2026-03-10

โฟลเดอร์นี้ใช้เก็บไฟล์ติดตาม Task งานที่ขอให้ AI ดำเนินการ

---

## รูปแบบไฟล์ Task

ทุก Task ใช้ชื่อไฟล์: **`TASK-{ชื่อย่อ}.md`**

```
doc/tasks/
├── README.md                  ← ไฟล์นี้ (คำอธิบายระบบ)
├── TASK-ui-redesign.md        ← Task: UI Redesign
├── TASK-backend-patterns.md   ← Task: Reusable Backend Patterns
└── TASK-{ชื่อย่อ}.md
```

---

## สถานะของ Task

| สถานะ | ความหมาย |
|-------|---------|
| `PENDING` | รอดำเนินการ (แผนเสร็จแล้ว รอยืนยัน) |
| `IN_PROGRESS` | กำลังดำเนินการอยู่ |
| `DONE` | เสร็จสมบูรณ์ |
| `CANCELLED` | ยกเลิก |

## Template ของไฟล์ Task

> **หลักการ**: Task file ต้องมีรายละเอียดเพียงพอให้ AI หรือผู้อ่านเข้าใจได้ทันทีว่า
> "ต้องทำอะไร ที่ไหน ทำไม และทำยังไง" โดยไม่ต้องถามเพิ่ม

```markdown
# TASK: {ชื่อ Task}

**สถานะ**: PENDING / IN_PROGRESS / DONE / CANCELLED
**วันที่เริ่ม**: YYYY-MM-DD
**วันที่เสร็จ**: - (กรอกเมื่อเสร็จ)

> สรุปทิศทาง / เป้าหมายหลักของ Task 1 บรรทัด
> เอกสารอ้างอิง: [ชื่อ](../path/to/doc.md)
```

## สิ่งที่ต้องมีในทุก Task file

| หัวข้อ | บังคับ? | คำอธิบาย |
|--------|---------|---------|
| Header (สถานะ/วันที่) | บังคับ | ทุก Task ต้องมีสถานะและวันที่เริ่ม |
| เป้าหมายหลัก | บังคับ | สรุป 1 บรรทัดว่า Task นี้ทำเพื่ออะไร |
| กฎเฉพาะ Task | ถ้ามี | กฎที่ต้องยึดถือตลอดการทำ Task นี้ |
| Dependencies | ถ้ามี | สิ่งที่ต้องเตรียมก่อนเริ่ม |
| Phase + Sub-task | บังคับ | แบ่งงานเป็น Phase → Sub-task ตามลำดับ |
| ปัญหาปัจจุบัน | บังคับ | ทุก sub-task ต้องระบุว่าปัจจุบันผิดยังไง |
| เป้าหมาย | บังคับ | ทุก sub-task ต้องระบุว่าจะเปลี่ยนเป็นอะไร |
| ลำดับการทำ | บังคับ | แนะนำลำดับที่ควรทำ พร้อมเหตุผล |

## กฎการใช้งาน

1. **ก่อนลงมือทำ**: AI จะสร้าง Plan ให้อ่านก่อนเสมอ
2. **หลังยืนยัน Plan**: AI สร้าง `TASK-{ชื่อ}.md` ในโฟลเดอร์นี้ ตามรูปแบบข้างบน
3. **ระหว่างทำงาน**: อัพเดต Task file ทุกครั้งที่ Phase/Sub-task เสร็จ
4. **เมื่อเสร็จสมบูรณ์**: เปลี่ยนสถานะเป็น `DONE` พร้อมบันทึกวันที่เสร็จ
5. **รายละเอียดต้องเพียงพอ**: ระบุ "ปัญหาปัจจุบัน" + "เป้าหมาย" ทุก sub-task — ห้ามเขียนแค่ checkbox
